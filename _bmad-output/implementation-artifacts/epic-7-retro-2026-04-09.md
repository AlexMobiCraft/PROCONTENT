# Ретроспектива Epic 7

## Метаданные

- Проект: PROCONTENT
- Эпик: 7
- Название: Rich Content Experience (Markdown Editor & Inline Media)
- Дата ретро: 2026-04-09
- Основание: `sprint-status.yaml`, stories 7.1/7.2, PRD, Architecture, deferred work, tech debt

## Краткий итог эпика

Epic 7 закрыл важный продуктовый разрыв между authoring и consumption слоями контента. В рамках эпика были внедрены rich-content authoring flow на базе Tiptap, отдельный pipeline для `inline-images`, безопасный рендер HTML-контента через DOMPurify и корректная композиция `gallery above text` в пользовательском просмотре поста.

С точки зрения результата эпик успешен: ключевые возможности доставлены, архитектурные границы в целом сохранены, regression coverage добавлен. Главный остаточный вывод не про недоделанную фичу, а про процесс: финальное состояние spread по нескольким артефактам несинхронно, и это уже организационный риск.

## Что сработало хорошо

- Разделение `gallery` и `editor` как независимых доменов состояния оказалось правильным решением и снизило риск деградации Story 4.1.
- Вынос editor-логики в отдельный feature slice `src/features/editor/*` удержал проект в существующей feature-driven архитектуре.
- Команда не пошла в опасный greenfield rewrite `PostForm`, а расширила brownfield flow эволюционно.
- Review loop реально улучшал результат, а не был формальностью: accessibility, selection preservation, hydration и link security были закрыты по ходу ревью.
- Для consumer-side rendering был выбран прагматичный путь: sanitize + HTML render вместо несоответствующего реальному формату `react-markdown`.
- Regression coverage стала частью delivery, а не послесловием: тесты добавлялись и для authoring, и для render path.

## Что было сложно

- На уровне planning и implementation была асимметрия терминов: в планировании фигурировал “Markdown”, а фактическое хранение после Story 7.1 стало HTML из Tiptap. Это потребовало корректировки технического решения уже по ходу Story 7.2.
- Rich-content flow оказался сквозной задачей через admin form, editor extensions, storage, rendering и tests. Для такого scope особенно критична синхронизация артефактов, а она осталась неполной.
- Часть важных исправлений выявлялась только после review и ручной проверки, а не на этапе первого implementation pass.
- В проекте уже накоплен соседний технический долг по scheduled publishing, Stripe и search; он не блокировал Epic 7 напрямую, но повышает общий fragility budget кодовой базы.

## Повторяющиеся паттерны

### Позитивные

- Сильный review-driven refinement: сначала deliverable, затем серия целевых доработок по безопасности, a11y и UX-пограничным случаям.
- Brownfield-safe thinking: решения принимались с учетом существующих flows, а не поверх них.
- Явное удержание Smart/Dumb boundaries в новых компонентах.

### Негативные

- Источник истины о статусе разъезжается между story-файлом и `sprint-status.yaml`.
- Существенные технические уточнения всплывают уже после старта implementation, а не фиксируются заранее в story/dev notes.
- Deferred items остаются документированными, но не всегда превращаются в отдельный входящий backlog с владельцем.

## Конкретные уроки

1. Для editor-driven фич нельзя оставлять неоднозначность формата хранения контента. Формат `posts.content` должен быть зафиксирован явно в PRD, Architecture и story до начала реализации.
2. Если story проходит через несколько review rounds, финальный статус должен обновляться в одном обязательном финальном шаге, иначе артефакты начинают противоречить друг другу.
3. Для DOM-sanitization и HTML rendering нужно заранее считать производительность и error containment отдельными concern-ами, а не только security concern-ами.
4. Deferred findings без явного владельца быстро превращаются в “известно, но не управляется”.

## Проверка предыдущих ретро

Предыдущее retrospective для Epic 6 в implementation artifacts не найдено. Непрерывность улучшений по эпикам сейчас не формализована.

Вывод:
- практика ретро есть смысл сделать обязательной хотя бы для завершенных больших эпиков;
- иначе полезные выводы остаются в отдельных story и review notes, но не переходят в системные правила команды.

## Риски и незавершенные хвосты

### Прямо связанные с Epic 7

- В story-файле `7-2-markdown-rendering-posts-inline-images-and-combined-layout.md` статус все еще `review`, тогда как в `sprint-status.yaml` story считается `done`. Это процессная несогласованность.
- В `deferred-work.md` для Story 7.2 зафиксированы два deferred пункта:
  - отсутствие оптимизации тяжелой sanitization-логики;
  - отсутствие Error Boundary для DOM-манипуляций.

### Системные хвосты, влияющие на следующий цикл

- Scheduled publishing содержит отложенные race-condition сценарии и вопросы надежности notification flow.
- В tech debt по Stripe остаются более серьезные инфраструктурные риски, чем те, что были в Epic 7.
- Search fixes уже зафиксировали проблемы с cache invalidation и тестовым покрытием, что говорит о повторяющемся классе “state correctness under change”.

## Оценка готовности результата

- Функциональная готовность Epic 7: высокая
- Архитектурная согласованность: хорошая, но с терминологическим дрейфом между “Markdown” и фактическим HTML
- Качество и тестируемость: хорошие
- Процессная завершенность: средняя из-за несинхронных статусов и deferred items без явного follow-up

Epic 7 можно считать завершенным по delivered scope, но не идеально закрытым по operational hygiene.

## Подготовка следующего цикла

Epic 8 в planning artifacts не обнаружен. Значит, следующее движение должно быть не “сразу писать новую story”, а сначала определить, что именно идет следующим:

- либо новый продуктовый scope;
- либо stabilization / cleanup sprint;
- либо целевой tech-debt batch по уже доставленным возможностям.

С учетом текущего состояния проекта более рационален короткий stabilization-first этап, а не немедленный старт новой крупной функциональности.

## Action Items

- Product / PM: зафиксировать в planning artifacts единый термин и формат хранения `posts.content` как HTML Tiptap output, чтобы будущие story не повторяли путаницу “Markdown vs HTML”.
- Scrum Master: добавить обязательный финальный checklist для story close-out: `story file status`, `sprint-status.yaml`, deferred findings, regression verification.
- Dev: выделить отдельную техническую story на stabilization `MarkdownRenderer`: производительность sanitization и стратегия error containment.
- Dev: разобрать deferred backlog и перенести пункты из `deferred-work.md` в нормализованный implementation backlog с явными владельцами.
- Team: принять правило “нет done без синхронизации артефактов”, даже если код уже смержен и тесты зеленые.

## Preparation Tasks перед следующим эпиком

- Синхронизировать финальный статус Story 7.2 во всех артефактах.
- Решить, нужен ли отдельный stabilization sprint перед новым продуктовым scope.
- Сформировать shortlist техдолга, который реально влияет на скорость следующего цикла: Stripe, scheduled publishing, search cache, MarkdownRenderer.
- Если стартует новый scope, сначала создать brief/plan с учетом урока про явную фиксацию data format и rendering contract.

## Ключевые выводы

1. Epic 7 доставил ценную функциональность без разрушения brownfield-архитектуры.
2. Review-процесс здесь был источником качества, а не тормозом.
3. Основная слабость спринта не в реализации, а в discipline layer: статусные артефакты и deferred backlog ведутся недостаточно строго.
4. Перед следующим большим эпиком проекту полезнее небольшой stabilization pass, чем немедленное расширение scope.
