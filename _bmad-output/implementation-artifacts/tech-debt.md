# Технический долг - PROCONTENT

## 💳 Процессинг Stripe и подписки

### [Story 1.5] Отложенные крайние случаи (Раунд 22)
- **Воскрешение "зомби-подписок" [Critical/High]**: В обработчике `handleInvoicePaymentSucceeded` запоздалый вебхук от *старой* подписки теоретически может перезаписать данные *новой* активной подписки в базе данных при сценарии отката на поиск по email. Требуется более надежный способ проверки хронологического порядка событий относительно состояния БД.
- **Логика оплаты возвращающихся пользователей [High]**: У возвращающихся пользователей с уже существующим `stripe_customer_id` события `checkout.session.completed` могут быть частично проигнорированы в пути поиска по email из-за строгой проверки `.is('stripe_customer_id', null)`. Эта проверка была добавлена как мера безопасности против подмены email (spoofing), но требует более гибкого подхода.
- **Скрытые сетевые ошибки Stripe API [High]**: Текущий блок `try/catch` вокруг `stripe.subscriptions.retrieve` поглощает сетевые ошибки. В худшем случае это может привести к отправке ответа 200 OK в Stripe без фактического обновления базы данных (событие будет пропущено). Необходимо разделять ошибки типа "Не найдено" (валидный сценарий) и "Ошибка соединения" (должна вызывать ошибку 5xx для повтора со стороны Stripe).
- **Глобальный лимит запросов для вебхуков [Medium]**: Текущий глобальный лимит 60 запр/мин может быть слишком жестким в периоды массового биллинга (например, первое число месяца). Рекомендуется перейти на более масштабируемый подход (например, Redis) или просто увеличить лимит после мониторинга нагрузки.

## 🛠️ Инфраструктура и безопасность
- **Кеширование криптографического ключа в Middleware [Low]**: Текущее глобальное кеширование ключа HMAC простое и эффективное для Edge-функций, но требует мониторинга на предмет утечек памяти или устаревания ключей при частой ротации секретов.
- **Защита от SQL-инъекций [Low]**: Хотя текущие запросы PostgREST санируются, будущие сложные фильтры с использованием `.or()` должны проходить тщательный аудит, чтобы исключить динамическую интерполяцию строк из недоверенных источников.
