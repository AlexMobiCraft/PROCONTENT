# Результаты ревью кода (Слепой Охотник / Blind Hunter)

1. **[MAINT] Отсутствует откат новых файлов при сбое обновления БД:** Функция `updatePost` загружает новые файлы в хранилище перед выполнением транзакций в БД. Если шаг 4 (`upsert` существующих медиа) или шаг 5 (`insert` новых медиа) завершается неудачно, пойманная ошибка выбрасывается снова без удаления недавно загруженных файлов, отслеживаемых в `uploadedUrls`. Это приведет к появлению "осиротевших" файлов в Supabase Storage. Доказательство: блок `catch` на строке 291 только перебрасывает ошибку без логики отката для `uploadedUrls`.

2. **[BUG] Возможное частичное сохранение файлов без записи в БД:** В `uploadFilesWithTracking`, если один батч завершается ошибкой, устанавливается и выбрасывается `firstError`, но ранее успешные батчи, чьи URL были добавлены в `uploadedUrls`, никогда не очищаются внутри самой функции. Если это происходит во время шага 3 в `updatePost`, `updatePost` также не перехватывает это корректно для очистки `uploadedUrls`. Доказательство: `throw firstError` в `uploadFilesWithTracking` после цикла.

3. **[SEC] Небезопасный fallback для генерации случайных UUID:** Fallback `generateUUID` для старых браузеров использует `Math.random()`, который криптографически небезопасен и предсказуем, что потенциально допускает коллизии UUID или угадывание путей в хранилище. Доказательство: `const r = (Math.random() * 16) | 0` в `uploadMedia.ts`.

4. **[BUG] Индекс порядка (order_index) может иметь пробелы или дубликаты:** В `updatePost` при удалении элементов оставшиеся элементы сохраняют свой исходный `order_index` без пересчета (например, 0, 2, 5). Это может сломать UI или логику сортировки, ожидающую непрерывные индексы. Доказательство: маппинг `existingItems` сохраняет `item.order_index` от клиента.

5. **[MAINT] Тихое проглатывание ошибок при откате:** В блоке `catch` функции `createPost` ошибка `removeStorageFiles` перехватывается и логируется в консоль через `console.warn`, но не оповещает систему и не повторяет попытку. Тихие сбои при откате — плохая практика. Доказательство: `catch((e) => { console.warn(...) })`.

6. **[PERF] Promise.allSettled скрывает индивидуальные ошибки до завершения батча:** В `uploadFilesWithTracking`, если загрузка файла завершается неудачно в начале батча, код все равно ждет завершения остальных файлов в батче перед выбрасыванием `firstError`, тратя пропускную способность и время. Доказательство: `await Promise.allSettled(batch.map(...))`

7. **[MAINT] Жестко закодированное имя бакета для хранилища:** `const STORAGE_BUCKET = 'post_media'` жестко закодировано в `uploadMedia.ts` вместо использования переменной окружения или конфига, что делает его негибким для разных сред (dev/staging/prod). Доказательство: Строка 4 в `uploadMedia.ts`.

8. **[BUG] Защитная проверка derivePostType скрывает неожиданные типы:** В `derivePostType` проверка `if (!hasImages && !hasVideos) return 'text'` предполагает наличие только валидных типов. Если `items` каким-то образом содержит неожиданный валидный тип (например, аудио), он некорректно по умолчанию установит тип поста `text` вместо отклонения или правильной обработки. Доказательство: логика `derivePostType`.

9. **[PERF] Отсутствуют заголовки кэширования файлов при загрузке:** Загрузка файла в `uploadSingleFile` не устанавливает заголовки управления кэшем (cache control). Поскольку это статические медиа-ассеты, они должны агрессивно кэшироваться браузером/CDN, но `cacheControl` отсутствует в опциях `storage.from().upload()`. Доказательство: вызов `supabase.storage.from(STORAGE_BUCKET).upload(path, file)` не имеет параметра options.

10. **[BUG] Отсутствует атомарная транзакция для `updatePost`:** Последовательность операций с БД в `updatePost` (обновление постов, upsert существующих медиа, insert новых медиа, удаление медиа) — это отдельные вызовы API. Сбой на середине оставляет базу данных в несогласованном состоянии, так как Supabase REST API не поддерживает транзакции для нескольких вызовов нативно. Доказательство: Несколько независимых вызовов `supabase.from()` в `updatePost`.
