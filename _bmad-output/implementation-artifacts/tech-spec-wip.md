# Tech-Spec: Перевод всего контента сайта на Словенский язык

**Created:** 2026-03-21T13:08:16+01:00

## Overview

### Problem Statement

Словенская аудитория сайта не имеет возможности читать контент и взаимодействовать с интерфейсом (кнопками, сообщениями, меню и т.д.) на родном языке. Текущие тексты, вероятно, захардкожены на одном языке (русском или английском), что делает сайт недоступным или неудобным для пользователей из Словении.

### Solution

Необходимо интегрировать надежное решение для интернационализации (i18n), подходящее для Next.js 16 App Router (предпочтительно `next-intl`), извлечь все захардкоженные строки интерфейса в локализационные словари (messages) и перевести их на словенский язык. Также потребуется добавить компонент для выбора и переключения языка пользователем.

### Scope

**In Scope:**
- Настройка `next-intl` для Next.js 16 App Router с поддержкой словенского (sl) и текущего базового языка (ru/en).
- Настройка роутинга через префикс в URL (например, `/sl/`).
- Словенский язык должен открываться по умолчанию при первом визите.
- Отключение автоматического определения локали по заголовкам браузера (`localeDetection: false`).
- Разделение всего статического текста (кнопки, лэйблы, заголовки, сообщения об ошибках) и замена его на механизмы перевода (`useTranslations`).
- Создание файлов словаря (JSON) для словенского языка.
- Разработка UI-компонента переключения языка (Dropdown Select) и его интеграция в навигационную панель рядом с профилем пользователя.
- Использование `src/proxy.ts` вместо `middleware.ts` согласно стандартам Next.js 16.

**Out of Scope:**
- Перевод динамического пользовательского контента, хранящегося в Supabase (посты, комментарии, профили пользователей).
- Автоматический машинный перевод пользовательского ввода.

## Context for Development

### Codebase Patterns

- Next.js 16 App Router - требуется обертка всего `app` в папку `[locale]/`.
- Использование `next-intl/navigation` оберток для работы с роутами (`Link`, `useRouter`, `usePathname`, `redirect`).
- **CRITICAL (Next.js 16):** Для контроля запросов (в том числе для i18n) должен использоваться `src/proxy.ts` (экспортирующий функцию `proxy`), а не устаревший файл `middleware.ts`. В `proxy.ts` необходимо совместить логику `next-intl` и логику аутентификации Supabase/закрытых роутов, а также исключить обработку `/api/*` путей.

### Files to Reference

| File | Purpose |
| ---- | ------- |
| `src/app/layout.tsx` | Основной layout, который переедет в `src/app/[locale]/layout.tsx` и будет обернут в i18n-провайдер |
| `next.config.mjs` | Настройка конфигурации плагина `@next/intl/plugin` |
| `src/proxy.ts` | Точка перехвата запросов для маршрутизации локалей (Next.js 16) |

### Technical Decisions

- **Роутинг:** Реализуем URL с префиксом локали (`/sl/`, `/en/`).
- **Точка входа (Middleware):** Логика редиректов и защита роутов должна базироваться в `src/proxy.ts`. Функция-обработчик должна называться `proxy`.
- **Default Locale:** `'sl'`
- **Locale Detection:** Отключено (`false`). Пользователи всегда сначала попадают на дефолтную локаль, потом могут сменить ее вручную с помощью Switcher.
- **Language Switcher UI:** Компактный выпадающий список (Dropdown Menu), расположенный в навигационной панели рядом с аватаром профиля.
- **Интеграция Supabase:** При внедрении префиксов нужно внимательно отследить, чтобы не сломались API-эндопоинты авторизации. В `proxy.ts` следует установить правило (matcher / exclusion) для `/api/auth/confirm` и других служебных роутов.
