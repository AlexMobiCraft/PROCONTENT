# Implementation Readiness Assessment Report

**Date:** 2026-04-01
**Project:** PROCONTENT — Отложенная публикация постов

## PRD Analysis

### Functional Requirements

FR1: Автор может выбрать режим публикации при создании поста — «Опубликовать сейчас» или «Запланировать»
FR2: Автор может указать дату и время плановой публикации в режиме «Запланировать»
FR3: Система отображает автору подтверждение планируемого времени публикации с указанием timezone
FR4: Система валидирует, что указанное время публикации находится в будущем
FR5: Автор может сохранить пост в статусе `scheduled` — пост не появляется в ленте участников
FR6: Автор может изменить содержимое запланированного поста до момента публикации
FR7: Автор может изменить запланированное время публикации до момента публикации
FR8: Автор может отменить запланированную публикацию — пост возвращается в статус `draft`
FR9: Автор может опубликовать запланированный пост немедленно, не дожидаясь назначенного времени
FR10: Система автоматически публикует посты со статусом `scheduled`, у которых наступило `scheduled_at`
FR11: Система публикует все пропущенные из-за downtime посты при следующем запуске планировщика
FR12: Система гарантирует публикацию каждого поста ровно один раз — повторные запуски не создают дублей
FR13: Система гарантирует отсутствие race condition при параллельных запусках планировщика
FR14: Автор может просматривать список всех запланированных постов с датами и статусами
FR15: Автор может перейти к редактированию любого запланированного поста из списка
FR16: Автор может отменить публикацию поста непосредственно из списка
FR17: Участники получают стандартное email-уведомление в момент автоматической публикации запланированного поста
FR18: Система не отправляет повторное email-уведомление при ретрае уже опубликованного поста

**Итого FRs: 18**

### Non-Functional Requirements

NFR1: Запланированный пост публикуется в течение 10 минут от `scheduled_at` (pg_cron 5 мин + overhead)
NFR2: Email-уведомление доставляется участникам в течение 5 минут после фактической публикации
NFR3: Cron-функция выполняется за ≤ 30 секунд при одновременной публикации нескольких постов
NFR4: Прямой вызов cron endpoint без авторизации возвращает `401`
NFR5: Поля `status` (`scheduled`) и `scheduled_at` изменяемы только ролью `admin` — RLS Supabase
NFR6: `published_at` устанавливается только системой (pg_cron) — клиентский API не может изменить
NFR7: Повторный запуск планировщика не изменяет статус поста с `published_at IS NOT NULL`
NFR8: При downtime до 24 часов все пропущенные публикации выполняются при восстановлении без вмешательства
NFR9: Ошибка публикации одного поста не блокирует публикацию остальных в той же итерации

**Итого NFRs: 9**

### Additional Requirements

- **Модель данных:** таблица `posts` расширяется полями `status TEXT DEFAULT 'draft'`, `scheduled_at TIMESTAMPTZ`, `published_at TIMESTAMPTZ` + индекс `idx_posts_scheduled WHERE status = 'scheduled'`
- **Cron:** pg_cron задача каждые 5 минут (Supabase PostgreSQL extension)
- **Timezone:** хранение в UTC, отображение через `Intl.DateTimeFormat` в браузерной timezone (CET/CEST)
- **Integration:** расширение существующей формы создания/редактирования поста (Story 4.1); существующий email-механизм (Resend) не требует переработки

PRD содержит все обязательные секции: Executive Summary, Success Criteria, Product Scope, User Journeys (3), Web App Requirements, Functional Requirements (18 FRs), Non-Functional Requirements (9 NFRs). Требования измеримы, capability-oriented (без implementation leakage), трассируемы к user journeys.
