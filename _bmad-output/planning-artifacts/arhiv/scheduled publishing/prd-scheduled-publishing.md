# Product Requirements Document — PROCONTENT: Отложенная публикация

**Author:** Alex
**Date:** 2026-04-01

## Executive Summary

**PROCONTENT Scheduled Publishing** — расширение существующей платформы закрытого клуба для создателей контента. Автор подготавливает пост заранее, назначает время публикации — пост появляется в ленте автоматически в назначенный момент, участники получают стандартное email-уведомление без ручного действия.

**Проблема:** Публикация привязана к присутствию автора онлайн. Контент выходит не тогда, когда аудитория активна, а тогда, когда у автора есть свободная минута. Невозможно подготовить серию постов заранее или запланировать выход на конкретный день недели.

**Решение:** Toggle «Запланировать» в существующей форме создания поста + pg_cron задача (каждые 5 минут) → автопубликация + стандартная email-рассылка без участия автора.

**Ключевое преимущество:** Фича встроена нативно в существующий процесс — не отдельный workflow, а дополнительный toggle в той же форме. Email-механизм уже работает, меняется только триггер (с INSERT на UPDATE статуса `published`). Минимальное техническое изменение, кардинально трансформирующее рабочий процесс: от реактивного режима к осознанному редакционному планированию.

**Классификация:** Web Application · Next.js 16 + Supabase · Brownfield · Complexity: Low-Medium · Планировщик: pg_cron (Vercel Hobby plan)

## Success Criteria

### User Success

Автор получает ценность, когда:
- **Планирует пост без присутствия онлайн** — пост опубликован в назначенное время без ручного действия
- **Завершает планирование за ≤ 3 дополнительных клика** относительно обычного флоу (toggle → datetime → submit)
- **Управляет расписанием** — редактирует и отменяет запланированные посты до публикации

### Business Success

- Доля новых постов через scheduling: **≥ 30% в первый месяц** после запуска

### Technical Success

- **0** дублирований email при одной публикации (идемпотентность через `published_at`)
- **0** пропущенных публикаций: пост публикуется в течение **10 минут** от `scheduled_at`
- **0** race conditions при параллельных cron-запусках (атомарный UPDATE + RETURNING)
- Пропущенные публикации (downtime) автоматически выполняются при следующем запуске планировщика

### Measurable Outcomes

| Метрика | Цель | Срок |
|---|---|---|
| Adoption: доля постов через scheduling | ≥ 30% | Месяц 1 |
| Email дублирование | 0 случаев | Всегда |
| Точность публикации | ≤ 10 мин от scheduled_at | Всегда |
| Время планирования | ≤ 3 дополнительных клика | День 1 |

## Product Scope

### MVP — Minimum Viable Product

**Подход:** Feature MVP — минимальное расширение, закрывающее конкретную боль (зависимость от присутствия онлайн). Монетизация не изменяется. Ресурсы: 1 full-stack разработчик.

- Статус `scheduled` в модели постов (`draft` → `scheduled` → `published`) + поля `scheduled_at`, `published_at`
- Toggle «Опубликовать сейчас / Запланировать» + datetime picker с timezone (CET/CEST) в форме создания/редактирования поста
- Validation: `scheduled_at` должен быть в будущем; preview «Пост будет опубликован [дата] в [время] [timezone]»
- pg_cron задача (каждые 5 минут) → атомарная публикация созревших постов → email-рассылка участникам
- Раздел «Запланировано» в admin-панели: таблица (заголовок, дата публикации, статус, кнопка отмены)
- Редактирование запланированного поста (смена времени, контента) и отмена (→ `draft`)

### Growth Features (Post-MVP)

- Уведомление автору о факте успешной автопубликации (email/in-app)
- Drag-and-drop календарный вид запланированных постов (если adoption высокий)

### Vision

Scheduled publishing MVP реализует редакционное планирование: автор создаёт очередь постов, распределяет по расписанию, управляет ритмом выхода контента. Расширение в сторону recurring-расписаний — при появлении спроса.

### Risk Mitigation

**Технический:** pg_cron новый для проекта → идемпотентность гарантирует отсутствие дублей даже при ошибках конфигурации.

**Market:** Adoption < 30% не критичен — «Опубликовать сейчас» остаётся дефолтным, фича не ломает текущий флоу.

**Resource:** Race condition → атомарный `UPDATE ... WHERE status='scheduled' AND published_at IS NULL RETURNING id`.

## User Journeys

### Journey 1 — Автор: Редакционное планирование (Happy Path)

**Персона:** Автор PROCONTENT. Создаёт контент пакетами: в воскресенье монтирует видео на всю неделю. Раньше приходилось садиться за компьютер каждый раз перед публикацией.

**Opening Scene:** Воскресенье, 16:00. Три готовых поста. Открывает admin-панель.

**Rising Action:** Создаёт первый пост как обычно. Вместо «Опубликовать» переключает toggle на «Запланировать». Datetime picker — выбирает понедельник 09:00. Под полем: «Пост будет опубликован 6 апреля в 09:00 (CET)». Повторяет для второго (среда 18:00) и третьего (пятница 10:00).

**Climax:** Раздел «Запланировано» — таблица с тремя постами. Понедельник 09:00:05 — pg_cron срабатывает, пост меняет статус на `published`, участницы получают email.

**Resolution:** Три поста — 20 минут в воскресенье вместо трёх отдельных сессий. Контент выходит точно по расписанию без участия автора.

**Capabilities revealed:** toggle, datetime picker, timezone preview, статус `scheduled`, таблица «Запланировано», автопубликация, email-рассылка.

---

### Journey 2 — Автор: Редактирование и отмена (Edge Case)

**Персона:** Тот же автор. Запланировал пост на пятницу 10:00 — в четверг пост потерял актуальность.

**Opening Scene:** Четверг вечером. Раздел «Запланировано».

**Rising Action:** Пост в таблице — `scheduled`, «5 апреля 10:00». Нажимает «Отменить публикацию» — пост возвращается в `draft`. Альтернативно: открывает форму редактирования, меняет `scheduled_at` на следующую неделю.

**Resolution:** Контроль работает в обе стороны — «запланировать» и «отменить без потери контента».

**Capabilities revealed:** отмена → `draft`, изменение `scheduled_at`, редактирование контента запланированного поста.

---

### Journey 3 — Система: Восстановление после downtime (System Edge Case)

**Opening Scene:** pg_cron не сработал в 09:00 из-за downtime Supabase. Пост остаётся в `scheduled`.

**Rising Action:** 09:05 — cron запускается. `UPDATE posts SET status='published', published_at=now() WHERE status='scheduled' AND scheduled_at <= now() RETURNING id`. Условие `<= now()` захватывает пропущенный пост.

**Resolution:** Пост опубликован в 09:05. `published_at` проставлен — повторный запуск в 09:10 не дублирует рассылку. 0 пропущенных публикаций, 0 дублей email.

**Capabilities revealed:** атомарный UPDATE, идемпотентность через `published_at IS NOT NULL`, самовосстановление при downtime.

## Web Application Specific Requirements

### Обзор

Все новые UI-элементы находятся в закрытой admin-зоне (за auth). Публичная часть приложения не затрагивается. SEO не применимо — все новые страницы за `noindex`.

### Browser Matrix

Наследуется от основного PRD. Datetime picker должен корректно работать на iOS Safari — нативный `<input type="datetime-local">` имеет специфику отображения на этой платформе.

### Responsive Design

Таблица «Запланировано»: горизонтальный скролл или сокращённые колонки на узких экранах. Datetime picker: полноэкранный на мобайле, inline на десктопе.

### Accessibility

Datetime picker: WCAG 2.1 Level AA — keyboard navigation, ARIA-атрибуты, touch target `min-h-[44px] min-w-[44px]`.

### Real-Time

Не требуется. Точность ±10 минут (pg_cron каждые 5 мин) — достаточно. Раздел «Запланировано» обновляется при перезагрузке страницы.

### Implementation Considerations

- **Timezone:** `scheduled_at` хранится в UTC, отображается в браузерной timezone через `Intl.DateTimeFormat`. Переход CET/CEST обрабатывается автоматически браузером.
- **Form integration:** toggle + conditionally rendered datetime picker встраиваются в существующую форму создания/редактирования поста (расширение Story 4.1).
- **Admin table:** отдельная страница/tab в существующей admin-панели (не модальное окно).

## Functional Requirements

### Post Scheduling

- FR1: Автор может выбрать режим публикации при создании поста — «Опубликовать сейчас» или «Запланировать»
- FR2: Автор может указать дату и время плановой публикации в режиме «Запланировать»
- FR3: Система отображает автору подтверждение планируемого времени публикации с указанием timezone
- FR4: Система валидирует, что указанное время публикации находится в будущем
- FR5: Автор может сохранить пост в статусе `scheduled` — пост не появляется в ленте участников

### Scheduled Post Management

- FR6: Автор может изменить содержимое запланированного поста до момента публикации
- FR7: Автор может изменить запланированное время публикации до момента публикации
- FR8: Автор может отменить запланированную публикацию — пост возвращается в статус `draft`
- FR9: Автор может опубликовать запланированный пост немедленно, не дожидаясь назначенного времени

### Automated Publication

- FR10: Система автоматически публикует посты со статусом `scheduled`, у которых наступило `scheduled_at`
- FR11: Система публикует все пропущенные из-за downtime посты при следующем запуске планировщика
- FR12: Система гарантирует публикацию каждого поста ровно один раз — повторные запуски не создают дублей
- FR13: Система гарантирует отсутствие race condition при параллельных запусках планировщика

### Admin Overview

- FR14: Автор может просматривать список всех запланированных постов с датами и статусами
- FR15: Автор может перейти к редактированию любого запланированного поста из списка
- FR16: Автор может отменить публикацию поста непосредственно из списка

### Notifications

- FR17: Участники получают стандартное email-уведомление в момент автоматической публикации запланированного поста
- FR18: Система не отправляет повторное email-уведомление при ретрае уже опубликованного поста

## Non-Functional Requirements

### Performance

- NFR1: Запланированный пост публикуется в течение **10 минут** от `scheduled_at` (pg_cron интервал 5 минут + overhead)
- NFR2: Email-уведомление доставляется участникам в течение **5 минут** после фактической публикации
- NFR3: Cron-функция выполняется за **≤ 30 секунд** при одновременной публикации нескольких постов

### Security

- NFR4: Прямой вызов cron endpoint без авторизации возвращает `401`
- NFR5: Поля `status` (`scheduled`) и `scheduled_at` изменяемы только ролью `admin` — RLS Supabase запрещает изменение участникам
- NFR6: `published_at` устанавливается только системой (pg_cron) — клиентский API не может изменить это поле

### Reliability

- NFR7: Повторный запуск планировщика не изменяет статус поста с `published_at IS NOT NULL` — идемпотентность гарантирована
- NFR8: При downtime до **24 часов** все пропущенные публикации выполняются автоматически при восстановлении без вмешательства администратора
- NFR9: Ошибка публикации одного поста (например, сбой email-провайдера) не блокирует публикацию остальных постов в той же итерации
