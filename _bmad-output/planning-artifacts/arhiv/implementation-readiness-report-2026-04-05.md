## PRD Analysis

### Functional Requirements

**Authentication, Subscription & Access Management**
- FR1 [M]: Участница может оформить платную подписку через Stripe (12,99€/мес или 34€/3 мес)
- FR1.1 [M]: При первичной регистрации участница получает ссылку на email, верифицирует почту и устанавливает пароль
- FR1.2 [M]: Участница может входить через email и пароль
- FR1.3 [M]: Участница может запросить сброс пароля на подтверждённый email
- FR2 [M]: Участница может отменить подписку в любое время через личный кабинет
- FR3 [M]: Система автоматически закрывает доступ при неоплате или отмене подписки
- FR4 [M]: Участница может просмотреть статус подписки (активна / истекает / отменена)
- FR5 [M]: Система восстанавливает доступ автоматически при успешной повторной оплате

**Landing & Conversion**
- FR6 [M]: Незарегистрированная посетительница может просмотреть превью-посты на лендинге без регистрации
- FR7 [M]: Незарегистрированная посетительница может прочитать отзывы участниц на лендинге
- FR8 [M]: Незарегистрированная посетительница может перейти к оплате напрямую с лендинга
- FR9 [S]: Автор может выбрать, какие посты отображаются как превью на лендинге

**Onboarding**
- FR10 [M]: Новая участница после первой оплаты попадает на специальную onboarding-страницу
- FR11 [M]: Участница видит на onboarding-странице подборку «Начни здесь» (топ-5 постов)
- FR12 [M]: Участница может перейти в WhatsApp-группу по ссылке с onboarding-страницы
- FR13 [S]: Автор может управлять содержимым onboarding-страницы

**Content Feed & Discovery**
- FR14 [M]: Участница может просматривать ленту постов в хронологическом порядке

- FR18 [S]: Участница может просмотреть Telegram-архив в хронологическом порядке

**Content Creation & Management**

### Non-Functional Requirements

**Performance**
- NFR1: LCP ≤ 2.5 сек на мобайле (3G)
- NFR2: TTI для публичных страниц ≤ 4 сек
- NFR3: Страницы с видео — TTI ≤ 4 сек, видео не блокирует интерактивность
- NFR4: Изображения загружаются ≤ 1 сек на мобайле (3G)
- NFR4.1: В галерее одновременно активно только одно видео (E2E-тест)
- NFR4.2: Все инлайн-изображения используют lazy loading (Lighthouse)
- NFR5: 50 одновременных пользователей, P95 ≤ 500ms для API

**Security**
- NFR6: TLS/HTTPS для всех страниц и API
- NFR7: Сессии ≤ 30 дней, инвалидация ≤ 60 сек после отмены/неоплаты
- NFR8: Stripe webhook — проверка цифровой подписи перед обработкой
- NFR9: Карточные данные хранит только Stripe
- NFR10: Страница Политики конфиденциальности до сбора данных
- NFR11: Cookie consent banner для незарегистрированных
- NFR12: Участница может запросить удаление аккаунта и данных
- NFR13: Данные удалённых участниц хранятся ≤ 3 месяца

**Accessibility**
- NFR14: WCAG 2.1 Level AA
- NFR15: Контраст текста ≥ 4.5:1
- NFR16: Alt-атрибуты для всех медиа
- NFR17: Keyboard navigation на десктопных браузерах

**Integration Reliability**
- NFR18: Обработка Stripe webhook идемпотентна
- NFR19: Сбои webhook логируются для ручной проверки
- NFR19.1: Таймаут Stripe API — дружелюбное сообщение, без двойных списаний
- NFR19.2: Telegram миграция — сохранение точки возобновления + exponential backoff
- NFR20: Email-уведомления доставляются ≤ 5 минут после публикации
- NFR21: Delivery rate ≥ 95%, мониторинг deliverability

**Data & Backup**
- NFR22: Резервирование БД ≥ 1 раз/сутки
- NFR23: Медиафайлы uptime ≥ 99.5%, скорость ≥ 1 Мб/с для Словении
- NFR24: Telegram-архив иммутабелен после импорта

**Scheduled Publishing Reliability**
- NFR25: Запланированный пост публикуется в течение 10 минут от `scheduled_at`
- NFR26: Каждый пост публикуется ровно один раз (идемпотентность через `published_at`)
- NFR27: Self-healing — пропущенные публикации выполняются при восстановлении планировщика

**Итого NFR: 31** (NFR1–NFR27, включая подпункты NFR4.1, NFR4.2, NFR19.1, NFR19.2)

### Missing Requirements

#### Критические пробелы (Must-Have — требуют новых Stories)

**FR16.2 [M] — Совместное отображение Markdown + инлайн-изображения + галерея**
- **Полный текст:** Контент поста поддерживает Markdown-рендеринг с инлайн-изображениями; если присутствуют и галерея, и инлайн-изображения — инлайн-текст отображается над сеткой галереи.
- **Воздействие:** Без этой истории разработчик реализует просмотр поста без логики совместного отображения — либо галерея перекроет инлайн-изображения, либо они задублируются.
- **Рекомендация:** Добавить AC в Story 2.6 или создать Story 2.6b с явными критериями: `react-markdown` рендеринг, поддержка `![](url)` синтаксиса, структура: блок текста с инлайн-контентом → блок `GalleryGrid` ниже.

**FR19.1 [M] — WYSIWYG-редактор для инлайн-изображений**
- **Полный текст:** Платформа предоставляет встроенный WYSIWYG-редактор (только текст + фото) для прямой заливки инлайн-изображений в безопасное облачное хранилище с автовставкой URL в тело поста.
- **Воздействие:** Story 4.1 описывает загрузку медиа только для галереи (`post_media`). Нет истории, описывающей загрузку изображений прямо в тело Markdown-текста поста через редактор.
- **Рекомендация:** Добавить Story 4.1b (или расширить 4.1) с AC: интеграция легковесного WYSIWYG (например, Tiptap или аналог), upload-обработчик в хранилище, автовставка URL-ссылки в Markdown, ограничения безопасности (тип файла, размер).

#### Частичное покрытие (требуют уточнения AC)

**FR19 [M] — Текст поста в Markdown**
- Story 4.1 описывает создание постов с медиагалереями, но не упоминает, что текстовое тело поддерживает Markdown-форматирование без лимитов на инлайн-изображения. Нужно добавить AC: «Текстовое поле поста принимает Markdown-разметку; количество инлайн-изображений в теле не ограничено».

**NFR4.2 — Lazy loading для инлайн-изображений (Lighthouse)**
- Story 2.3 реализует `LazyMediaWrapper` для медиа в карточках ленты. NFR4.2 требует lazy loading конкретно для инлайн-изображений внутри Markdown-тела поста. Нужно добавить AC в Story 2.6 или 2.3.

#### Структурное несоответствие (нумерация)

**FR36–FR42 (PRD) ↔ FR6.1–FR6.18 (Эпики)**
- PRD нумерует требования Scheduled Publishing как FR36–FR42 (7 FRs).
- Эпики используют собственную нумерацию FR6.1–FR6.18 (18 FRs более детализированных).
- Требования фактически эквивалентны, но несоответствие нумерации затрудняет трассировку.
- **Рекомендация:** Обновить раздел Requirements Inventory в epics.md, включив FR36–FR42 из PRD с примечанием о соответствии Epic 6 FR6.x, или наоборот — обновить Coverage Map.

### Story Quality Assessment — Violations

#### 🔴 Критические нарушения

**[C-1] FR16.2 — Отсутствует история или AC для Markdown + инлайн-изображения**
- **Нарушение:** Story 2.6 (Детальный просмотр поста) содержит AC «интерфейс корректно отрисовывает контент в зависимости от типа (Rich Text, Video Player, GalleryGrid)», но НЕ содержит:
  - AC для рендеринга Markdown с инлайн-изображениями
  - AC для комбинированного отображения (inline-контент над GalleryGrid)
  - Упоминания компонента `MarkdownRenderer`
- **Воздействие:** Разработчик реализует Story 2.6 без явных требований к Markdown-рендерингу — риск пропуска или некорректной реализации FR16.2.
- **Устранение:** Добавить в Story 2.6 AC: «Given пост с Markdown-содержимым и инлайн-изображениями; When открывается страница поста; Then компонент MarkdownRenderer рендерит текст с инлайн-изображениями (loading="lazy"); And если у поста есть галерея, блок GalleryGrid отображается ниже текстового блока».

**[C-2] FR19.1 — Отсутствует история для WYSIWYG-редактора инлайн-изображений**
- **Нарушение:** Story 4.1 описывает создание поста через загрузку медиа в галерею (`post_media`, Supabase Storage), но НЕ содержит:
  - AC для WYSIWYG-редактора в текстовом поле
  - AC для drag & drop / paste инлайн-изображений в тело поста
  - AC для загрузки в bucket `inline-images` с автовставкой URL в markdown
- **Воздействие:** Разработчик реализует форму поста без WYSIWYG — FR19.1 [M] не будет реализован.
- **Устранение:** Создать Story 4.1b «WYSIWYG-редактор для инлайн-изображений» или расширить Story 4.1 с AC для Tiptap-редактора.

#### 🟠 Значительные проблемы

**[M-1] Story 4.1 — Неполное описание текстового контента**
- **Нарушение:** Story 4.1 описывает загрузку медиафайлов, но не содержит AC для Markdown-форматирования в текстовом поле. Согласно FR19 «Автор может создавать посты с... текстовым контентом в формате Markdown без жёстких лимитов на инлайн-изображения».
- **Устранение:** Добавить AC: «Текстовое поле поста принимает Markdown-разметку; предпросмотр отображает форматированный текст».

**[M-2] Story 2.1 — Актор «разработчик» вместо пользователя**
- **Нарушение:** «As a разработчик, I want создать нормализованную таблицу `post_media`...» — технический актор вместо пользовательского.
- **Смягчение:** Это Database Schema story — широко принятая практика для Greenfield проектов. Однако нарушает принцип user-centric историй.
- **Устранение (опционально):** Переформулировать как «As a участница, I want видеть все медиафайлы поста в правильном порядке, So that могу просматривать галерею без артефактов».

**[M-3] Story 6.1 — Технический AC без пользовательского исхода**
- **Нарушение:** «As a автор» — но все AC сводятся к SQL-миграции, RLS-политикам и `supabase gen types`. Пользователь не может ощутить ценность этой истории напрямую.
- **Смягчение:** Schema stories для Scheduled Publishing неизбежно технические — допустимо как основание для Epic 6.
- **Устранение (опционально):** Переформулировать как «As a автор, I want платформа надёжно отслеживала статус каждого поста, So that я знала, что запланированные посты выйдут точно в срок».

**[M-4] NFR4.2 — Отсутствует AC в историях**
- **Нарушение:** NFR4.2 (lazy loading для инлайн-изображений) задокументирован в PRD и архитектуре, но ни одна история не содержит явного AC для проверки этого требования через Lighthouse.
- **Устранение:** Добавить AC в Story 2.6: «And все `<img>` в MarkdownRenderer имеют атрибут `loading="lazy"` (проверяется через Lighthouse audit)».
