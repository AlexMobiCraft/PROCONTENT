# Исправление проблемы Supabase Magic Link "Verify requires a verification type"

Это известная проблема Supabase (а точнее механизма PKCE-авторизации) при использовании `{{ .ConfirmationURL }}` в Email-шаблонах: иногда из-за почтовых клиентов, настроек отправки, или встроенного Go-шаблонизатора Supabase, параметры URL оказываются жестко закодированы (URL-encoded).

Если присмотреться к вашей ссылке, часть строки параметров была слита в один ключ:
`?token%3Dpkce_60ca...%26type%3Dmagiclink%26redirect_to...`

Поскольку символы `=` и `&` были закодированы как `%3D` и `%26`, родной сервер авторизации Supabase (на который ведет ссылка) **не находит параметр `type`** (то есть он видит один большой непонятный параметр `token%3D...`) и возвращает ошибку `code: 400`, `msg: "Verify requires a verification type"`.

### 🛠️ Как это исправить

Согласно архитектурным рекомендациям для связки **Next.js App Router (SSR) + PKCE + Supabase**, самым корректным и безопасным решением является полный **отказ от промежуточного перенаправления серверами Supabase (`/auth/v1/verify...`)**. 
Вместо этого мы берем подтверждение на себя и направляем пользователя из почты напрямую на наш сайт.

Для этого в проект уже добавлен необходимый Route Handler (файл `src/app/auth/confirm/route.ts`). Он умеет парсить `token_hash` из URL, обменивать его на рабочую сессию на сервере и перенаправлять пользователя в `/feed`.

### ⚙️ Инструкция по настройке Supabase Dashboard

Поскольку шаблоны писем живут и настраиваются не в коде, а в **Панели управления Supabase**, вам нужно зайти туда и отредактировать письмо.

1. Зайдите в проект на **[Supabase Dashboard](https://supabase.com/dashboard)**.
2. Перейдите в раздел **Authentication** -> **Email Templates**.
3. Откройте вкладку **Magic Link** (и "Confirm Signup", если вы используете подтверждение email при регистрации).
4. Замените ссылку в шаблоне (скорее всего сейчас там написано `href="{{ .ConfirmationURL }}"`) на нашу кастомную ссылку. 

**Для HTML-кода шаблона замените тег `<a>` на следующий:**
```html
<a href="{{ .SiteURL }}/auth/confirm?token_hash={{ .TokenHash }}&type=email">Войти в Procontent</a>
```

*(Если письмо отдается еще и в Plain Text, то соответственно замените `{{ .ConfirmationURL }}` на `{{ .SiteURL }}/auth/confirm?token_hash={{ .TokenHash }}&type=email`)*

**Почему это решит проблему?**
1. Ссылка больше не будет содержать никаких непонятных кодировок (параметры передаем мы сами).
2. Ссылка будет вести сразу на наш Next.js (`http://localhost:3000/auth/confirm` или продакшен), а не на домен Supabase. 
3. Наш новый Route Handler `auth/confirm` корректно расшифрует токен, установит secure-cookie и переключит на нужную страницу без ошибок сервера.

*(Обратите внимание: форма ввода 6-значного OTP-кода, при этом, не пострадала от бага и продолжит работать точно так же)*
